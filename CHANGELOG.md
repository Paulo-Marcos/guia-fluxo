## [Unreleased]

### Fixed
- **`finish`/commit: deleção de arquivo e atomicidade status↔commit (D-081).** Dois defeitos que morderam no finish do D-077. (1) `git_commit` fazia `git add -- <files>`, que falha com `pathspec did not match any files` quando um arquivo da task foi deletado, abortando o commit — trocado por `git add -A -- <files>`, que reconcilia o index com o working tree e cobre adições, modificações **e** deleções no mesmo stage. (2) `cmd_finish` persistia o status `Validada` (`save_task`) **antes** de commitar; se o commit estourava (pathspec, hook de lock, staged inesperado), a task ficava `Validada` sem nenhum commit por trás. Agora o status pré-finish é guardado e, no erro do commit, revertido (status + `save_task`/`set_current`/`upsert`) antes de propagar a exceção — `Validada` só persiste se o commit suceder. Escolhida a abordagem de rollback (não "commitar antes de salvar") para o commit final continuar enxergando o estado `.guia/` já gravado. Testes em `tests/test_finish_commit.py` (2/2: deleção commitada + commit que falha não deixa `Validada`).

## [0.4.0] - 2026-06-22

### Added
- **Hierarquia epic → stories (D-049).** Novo kind `epic` com prefixo próprio `E-NNN` (numeração independente de `D-NNN` — leitura instantânea de épico vs story). Verbo novo `guia epic "..."` cria o orquestrador; flag `--under E-NNN` em `feature/bug/chore` cria a story filha (recusa parent inexistente ou não-épico). `status E-NNN` imprime árvore agregada (`Progresso: closed/total` + lista de filhos + aviso de bloqueio); `finish E-NNN` é **recusado** enquanto qualquer filho estiver em status não-terminal (reusa `STATUSES_SATISFY_DEPENDENCY` do D-067 — `Validada`/`Finalizada`/`Resolvida`/`Cancelada` contam como terminais). Hierarquia de 2 níveis (sem epics aninhados); `parentId` imutável; `cancel E-NNN` NÃO cascateia para filhos. Cross-tool via manifest → `/guia:epic` no Claude + skill em `.agents/skills/`. Testes em `tests/test_epic.py` (10/10).
- **Dependência entre demandas (D-067).** Campo `dependsOn: [D-XYZ, ...]` por task. `start` e `promote` ficam recusados enquanto qualquer dependência não atingir status terminal (`Validada`/`Finalizada`/`Resolvida`/`Cancelada`). `ready`/`finish` ficam livres. Cancelada também satisfaz (dep encerrada explicitamente não deve travar a dependente). Superfície: `--depends-on D-X` (repetível) em `feature/bug/chore`; novo subcomando dedicado `depends add/remove/list <id> [--on D-X]`. Recusa auto-dependência, id inexistente em `tasks.json`, e qualquer dep que crie ciclo (DFS sobre o grafo). Novo módulo `core/src/_cli_deps.py`. Testes em `tests/test_depends.py` (11/11).
- **`guia upgrade` — migração automática de layout (D-091).** Verbo determinístico que migra projetos existentes do layout antigo (`FEATURES.md` na raiz, `features/registry.yaml`, `features/lock-ignore.txt`) para o atual (`.guia/DEMANDAS.md`, `.guia/locks/registry.yaml`, `.guia/locks/lock-ignore.txt`); remove `features/` se ficar vazio. Usa `git mv` quando o arquivo está rastreado (preserva histórico de rename) e cai para `shutil.move` quando não dá. Idempotente: NOOP quando não há nada a mover; `--dry-run` lista o plano sem mutar; recusa (exit 1) se algum destino já existe (resolva à mão). Adicionado a `_NO_AUTO_INIT` (não auto-cria `.guia/` antes de migrar). Distinto de `init` (setup virgem), de `/plugin update` (atualiza o plugin) e de `backlog migrate` (B-NNN legacy). Cross-tool via manifest → `/guia:upgrade`. Testes em `tests/test_upgrade.py` (5/5).
- **Subcomando `backlog resolve` (D-079).** `guia backlog resolve <id> [--reason]` retira do backlog ativo um item já entregue por outra demanda (ou obsoleto) **sem** promover: marca `status=Resolvida` + `resolvedAt` (+ `resolution` quando há `--reason`), preservando o item no arquivo para histórico (soft-resolve, idempotente). Funciona nas duas fontes que `backlog list` une (`D-NNN` em `tasks.json` e `B-NNN` legacy em `backlog.json`); resolvidos somem da `list`. Fechou B-009/B-011/B-017 (marketplace remoto, `cancel`, `plan` já entregues). Testes em `tests/test_backlog_resolve.py` (4/4).
- **`status --all` (B-014).** Quadro das tasks `Em desenvolvimento`, marcando a `current`. Quando há 2+ ativas ao mesmo tempo, avisa sobre a ambiguidade do `current-task.json` global (B-018) — comandos sem id explícito podem pegar a task errada. Testes em `tests/test_status_all.py` (3/3).

### Decisões arquiteturais
- **ADR-0016 — camada de serviços (`guia service`) RECUSADA.** Considerada e recusada após análise: a camada seria uma indireção de registry sobre skills, sem ganho funcional. Skills (Agent Skills / `SKILL.md`) já fazem o trabalho — orquestram outras skills, recebem alvo, carregam critério, produzem saída — e o catálogo de skills do consumidor (ou um plugin próprio) já é a personalização. A analogia com locks (que justificava o CRUD) não se sustenta: skills não são enforcement-driven; editar um `SKILL.md` à mão é seguro. Cross-tool também já está coberto por `SKILL.md` nativamente. Caminho adotado: skills + plugin próprio do usuário. D-062..066 reativadas como D-085–D-089 ("construir como skills"); D-084 (implementação) cancelada.
- **ADR-0017 — manter `core/src/` flat; D-053 RECUSADA.** A proposta era reorganizar os 19 arquivos flat em quatro subpastas (`cli/`/`domain/`/`infra/`/`shared/`). Avaliação pela skill `proposal-architect`: pelos drivers atuais (single-user CLI, domínio pequeno e estável, sem trocas de infra previstas, time de 1 dev), o projeto está no Degrau 1 da escada de complexidade. Subir para Degrau 2 (Clean/Hexagonal) seria over-engineering nominal — a convenção de prefixo já existente (`_cli_*`, `_locks`, `_features_md`, `_constants`) já agrupa por camada lógica implícita. ADR-0017 formaliza convenção + regras de coesão e fixa gatilho de revisão (~30 arquivos, troca de infra, 2+ devs novos, ou arquivo `_cli_*` > 500 linhas).
